

## Fix: Restore Data Failing Due to Missing INSERT Policies

### Problem
The restore function fails because **7 tables** are missing INSERT RLS policies, and the recently added `expense_accounts` INSERT policy doesn't account for Cipher (super admin) users who aren't formal company members.

The database logs confirm:
- `new row violates row-level security policy for table "revenue_sources"`
- `new row violates row-level security policy for table "expense_accounts"`

### Tables Missing INSERT Policies
1. `revenue_sources`
2. `allocations`
3. `khata_transfers`
4. `monthly_fee_history`
5. `student_siblings`
6. `courses`

### Table With Broken INSERT Policy
7. `expense_accounts` -- current policy requires `is_company_member()` which fails for Cipher users doing cross-company operations

### Solution
Create a single database migration that:

1. **Drops** the existing `expense_accounts` INSERT policy
2. **Creates INSERT policies** for all 7 tables using a pattern that supports both regular members AND Cipher users:

```sql
WITH CHECK (
  company_id = get_active_company_id(auth.uid())
  AND (
    is_company_member(auth.uid(), company_id)
    OR is_cipher(auth.uid())
  )
)
```

### Technical Details

Each policy will use the appropriate permission function where one already exists:
- `courses`: admin or cipher
- `revenue_sources`: admin or cipher (config-level table)
- `allocations`: member or cipher (auto-generated by triggers, but restore needs manual insert)
- `khata_transfers`: `company_can_transfer()` or cipher
- `monthly_fee_history`: member or cipher
- `student_siblings`: member or cipher
- `expense_accounts`: member or cipher (re-created with cipher support)

No frontend code changes are needed -- the restore logic in `useDataManagement.ts` is correct; it just needs the database to allow the inserts.
